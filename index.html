一、前言

在資訊科學領域中，排序 (Sorting) 是最基礎且最重要的運算之一。無論是搜尋引擎的資料檢索、資料庫的索引建立，還是社群媒體的動態排序，都離不開高效能的排序演算法。

本報告針對五種經典排序演算法進行深度探討。我們將從演算法的運作邏輯、時間與空間複雜度出發，並透過 JavaScript 撰寫自動化測試腳本，實測在不同數據規模（n）下各演算法的反應時間。最終目的是理解在處理海量數據時，如何選擇最適合的工具以達到效能最佳化。

二、排序法原理詳細介紹

排序法名稱 核心運作原理 操作連結
氣泡排序 (Bubble Sort) 最直觀的演算法。重複走訪序列，每次比較相鄰的兩個元素，若順序錯誤則對調。每一輪會將目前未排序區的最大值「浮」到序列末端。 進入模擬
選擇排序 (Selection Sort) 在未排序序列中找到最小（大）元素，存放到排序序列的起始位置，再從剩餘未排序元素中繼續尋找，重複直到排序完成。特點是交換次數少。 進入模擬
插入排序 (Insertion Sort) 類似整理撲克牌。將資料分為「已排序」與「未排序」兩部分，逐一取出未排序元素，在已排序序列中從後往前掃描，找到位置並插入。 進入模擬
合併排序 (Merge Sort) 採用分而治之 (Divide and Conquer)。將大陣列遞迴分割成最小單位（單一元素），再兩兩按順序合併。優點是性能極其穩定。 進入模擬
快速排序 (Quick Sort) 選取一基準值 (Pivot)，將序列分為比 Pivot 小與比 Pivot 大的兩部分，再對這兩部分遞迴排序。是實務上平均速度最快的演算法。 進入模擬

五、心得與結論

1. 效能觀察報告

  • 小數據規模： 當 n 較小（如 < 1000）時，五種演算法的體感差異極小。有趣的是，插入排序在處理「近乎排序完成」的資料時，速度甚至會快過合併排序，因為它的係數較小且不需額外記憶體。
  • 大數據規模： 當 n 提升至 10,000 以上時，O(n²) 類型的演算法（氣泡、選擇）會出現明顯的「效能懸崖」。執行時間會隨資料量平方增長，導致瀏覽器可能出現暫時無回應的情況。
  • 穩定效能組： 合併排序與快速排序展現了 O(n log n) 的驚人效率，處理 10,000 筆資料通常能在 10ms 內完成。

2. 綜合結論與建議

根據理論分析與實驗數據，我們可以得出以下結論：

  • 穩定性考量： 若需要保持相同數值的原始相對順序，應選擇合併排序或插入排序。
  • 空間受限： 快速排序雖然快，但遞迴需要堆疊空間；若裝置記憶體極度受限，選擇排序或插入排序（原地排序）是較好的選擇。
  • 通用的最優解： 在現代開發（如處理 API 回傳的巨量資料）中，應優先選擇快速排序 (Quick Sort)，這也是許多內建函式庫（如 JavaScript 的 sort()）底層實作的核心想法。
